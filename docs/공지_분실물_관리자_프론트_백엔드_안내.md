# 공지사항·분실물 관리자 페이지 — 프론트 개발 시 참고할 백엔드 사항

관리자 페이지(공지사항, 분실물) 프론트 개발 시 알아야 할 백엔드 정보만 정리했습니다.

---







## 1. 공통 (인증·Base URL·CORS)

- **Base URL**  
  - 개발: `http://localhost:8080`  
  - 배포 시 백엔드 서버 URL로 변경

- **관리자 API 인증**  
  - **공지·분실물 관련 API는 모두 인증 필요.**  
  - 요청 시 헤더에 **Access Token** 필수:  
    `Authorization: Bearer {accessToken}`  
  - 토큰 없거나 만료 시 **401 Unauthorized**

- **로그인·토큰**  
  - 로그인: `POST /auth/login` (body: `studentNumber`, `password`)  
  - 응답 body에 `accessToken`, Set-Cookie로 `refreshToken` (HttpOnly)  
  - 재발급: `POST /auth/reissue` (쿠키만 보내면 됨, `credentials: 'include'` 필수)  
  - 로그아웃: `POST /auth/logout` (쿠키 전송 후 호출)

- **CORS**  
  - 허용 origin: 설정된 프론트 도메인 (예: localhost:3000, 5173)  
  - **모든 API 요청에 `credentials: 'include'`(또는 axios `withCredentials: true)` 설정해야 쿠키 전송됨.**

- **에러 응답 형식 (공통)**  
  - JSON: `{ "message": "에러 메시지", "status": 400|401|404|500, "errors": null 또는 [{ "field": "필드명", "reason": "메시지" }] }`  
  - 검증 실패(400): `errors` 배열에 필드별 메시지.

---

## 2. 공지사항 API (구현됨)

관리자 공지 관련 API는 **모두 `/api/admin` 아래**이며, **모두 인증 필요.**

### 2-1. 두 가지 “공지” 개념

- **긴급 공지 (홈 화면 한 줄)**  
  - **엔드포인트**: `GET/PUT /api/admin/emergency`  
  - **역할**: 홈 화면 상단에 뜨는 **한 줄 메시지** 1개만 존재.  
  - **데이터**: `message`, `isActive` 등. (별도 테이블, 레코드 1개)

- **일반 공지 + 맨 위 고정(핀)**  
  - **엔드포인트**: `GET/POST/PUT/DELETE /api/admin/notices`  
  - **역할**: 공지 **목록**을 보여주는 게시판.  
  - **`isEmergency`**: 그중 **한 개**를 “맨 위 고정”할 때 사용.  
  - **데이터**: `title`, `content`, `author`, `isEmergency`, `isActive` 등.  
  - **정렬**: 백엔드는 `createdAt` 내림차순. **맨 위 고정**은 `isEmergency === true`인 항목을 프론트에서 리스트 맨 위에 배치하면 됨.

---

### 2-2. 긴급 공지 (홈 한 줄)

| 메서드 | 경로 | 설명 | Request | Response |
|--------|------|------|---------|----------|
| GET | `/api/admin/emergency` | 긴급 공지 1개 조회 | 없음 | 아래 EmergencyNoticeResponse |
| PUT | `/api/admin/emergency` | 긴급 공지 수정 | 아래 UpdateEmergencyRequest | EmergencyNoticeResponse |

**UpdateEmergencyRequest (PUT body)**  
- `message` (string, optional): 한 줄 메시지  
- `isActive` (boolean, optional): 사용 여부  

**EmergencyNoticeResponse**  
- `id` (number)  
- `message` (string)  
- `isActive` (boolean)  
- `createdAt`, `updatedAt` (ISO 8601 문자열)

---

### 2-3. 일반 공지 (목록·생성·수정·삭제)

| 메서드 | 경로 | 설명 | Request | Response |
|--------|------|------|---------|----------|
| GET | `/api/admin/notices` | 목록 조회 (검색·페이지네이션) | Query: `keyword`, `page`, `size` | Spring Page (아래 참고) |
| POST | `/api/admin/notices` | 공지 생성 | CreateNoticeRequest | NoticeResponse |
| PUT | `/api/admin/notices/{id}` | 공지 수정 | UpdateNoticeRequest | NoticeResponse |
| DELETE | `/api/admin/notices/{id}` | 공지 삭제 (소프트 삭제) | 없음 | 204 No Content, body 없음 |

**Query (GET /api/admin/notices)**  
- `keyword` (optional): 제목 검색  
- `page` (optional): 0부터 시작하는 페이지 번호  
- `size` (optional): 한 페이지 크기 (기본 10)

**CreateNoticeRequest (POST body)**  
- `title` (string, 필수)  
- `content` (string, 필수)  
- `author` (string, 필수)  
- `isEmergency` (boolean, optional, 기본 false) — true면 이 공지를 “맨 위 고정”. 기존에 true였던 공지는 백엔드에서 자동으로 false 처리.

**UpdateNoticeRequest (PUT body)**  
- `title` (string, 필수)  
- `content` (string, 필수)  
- `author` (string, 필수)  
- `isEmergency` (boolean, optional)

**NoticeResponse (단일 공지)**  
- `id`, `title`, `content`, `author`  
- `isEmergency`, `isActive`  
- `createdAt`, `updatedAt` (ISO 8601)

**목록 응답 (GET /api/admin/notices) — Spring Page**  
- `content`: NoticeResponse[]  
- `totalElements`: 전체 개수  
- `totalPages`: 전체 페이지 수  
- `size`: 요청한 size  
- `number`: 현재 페이지 번호 (0부터)  
- `first`, `last`: 첫/마지막 페이지 여부  
- 기타 `numberOfElements`, `empty` 등 필요 시 사용

**삭제 동작**  
- DELETE는 **소프트 삭제**. DB에서 지우지 않고 `isActive=false`로만 변경.  
- 목록 API는 **isActive=true 인 공지만** 반환하므로, 삭제한 공지는 목록에 안 나옴.

---

## 3. 분실물 API (구현됨)

관리자 분실물 관련 API는 **모두 `/api/admin/lost-items` 아래**이며, **모두 인증 필요.**

### 3-1. 분실물 데이터 구조

한 행(분실물 1개)에 담기는 정보:

- **id**: number — PK  
- **itemName**: string — 물건 이름 (예: `아이폰 13 Pro (검정)`)  
- **imageUrl**: string — 이미지 URL (썸네일, 선택)  
- **foundLocation**: string — 습득 장소 (예: `중앙 무대 앞`)  
- **foundDate**: string — 습득 날짜, 형식 `yyyy-MM-dd` (예: `2023-05-14`)  
- **isClaimed**: boolean — 인수 여부 (미인수=false, 인수완료=true)  
- **receiverName**: string|null — 인수자 이름 (isClaimed=true일 때만 의미)  
- **receiverNote**: string|null — 인수 관련 간단 메모  
- **isActive**: boolean — 소프트 삭제 여부 (true만 목록에 노출)  
- **createdAt / updatedAt**: string — ISO 8601 날짜시간 (`2026-03-02T12:34:56` 형식)

---

### 3-2. 목록 조회 (전체 / 미인수 / 인수완료)

| 메서드 | 경로 | 설명 | Request | Response |
|--------|------|------|---------|----------|
| GET | `/api/admin/lost-items` | 분실물 목록 조회 | Query: `status`, `page`, `size` | Spring Page (아래 참고) |

**Query (GET /api/admin/lost-items)**  
- `status` (optional): `ALL` \| `UNCLAIMED` \| `CLAIMED`  
  - 미전달 또는 `ALL`: 전체 (`isActive=true`)  
  - `UNCLAIMED`: 미인수만 (`isClaimed=false`)  
  - `CLAIMED`: 인수완료만 (`isClaimed=true`)  
- `page` (optional): 0부터 시작하는 페이지 번호  
- `size` (optional): 한 페이지 크기 (기본 10)

**목록 응답 예 (content 원소)**  

```json
{
  "id": 10,
  "itemName": "아이폰 13 Pro (검정)",
  "imageUrl": "https://.../iphone.jpg",
  "foundLocation": "중앙 무대 앞",
  "foundDate": "2023-05-14",
  "isClaimed": false,
  "receiverName": null,
  "receiverNote": null,
  "isActive": true,
  "createdAt": "2026-03-02T12:34:56",
  "updatedAt": "2026-03-02T12:34:56"
}
```

**목록 응답 구조 — Spring Page**  
- `content`: LostItemResponse[] (위와 같은 구조)  
- `totalElements`: 전체 개수  
- `totalPages`: 전체 페이지 수  
- `size`: 요청한 size  
- `number`: 현재 페이지 번호 (0부터)  
- `first`, `last`: 첫/마지막 페이지 여부  
- 기타 `numberOfElements`, `empty` 등 필요 시 사용

---

### 3-3. 분실물 등록

| 메서드 | 경로 | 설명 | Request | Response |
|--------|------|------|---------|----------|
| POST | `/api/admin/lost-items` | 분실물 등록 | CreateLostItemRequest | LostItemResponse |

**CreateLostItemRequest (POST body)**  

```json
{
  "itemName": "에어팟 프로 케이스 (흰색)",
  "imageUrl": "https://.../airpods-case.png",
  "foundLocation": "푸드트럭 존",
  "foundDate": "2023-05-14",
  "isClaimed": false,
  "receiverName": null,
  "receiverNote": null
}
```

- `foundDate`: 문자열 `"yyyy-MM-dd"` 포맷  
- `isClaimed` 생략 시 기본값 false  
- 성공 시 body에 `LostItemResponse` 반환

---

### 3-4. 분실물 수정 (인수 여부 + 인수자/메모)

| 메서드 | 경로 | 설명 | Request | Response |
|--------|------|------|---------|----------|
| PUT | `/api/admin/lost-items/{id}` | 분실물 수정 | UpdateLostItemRequest | LostItemResponse |

**UpdateLostItemRequest (PUT body)** — 생성과 같은 필드 사용

```json
{
  "itemName": "검은색 가죽 지갑",
  "imageUrl": "https://.../wallet.png",
  "foundLocation": "잔디광장",
  "foundDate": "2023-05-13",
  "isClaimed": true,
  "receiverName": "홍길동",
  "receiverNote": "학생증 확인 후 인계 완료"
}
```

동작 규칙:

- `isClaimed=true`일 때  
  - `receiverName`, `receiverNote` 값이 그대로 저장됨 → 인수완료 + 메모 표시용
- `isClaimed=false`로 변경할 때  
  - 백엔드에서 `receiverName`, `receiverNote`를 `null` 로 초기화 → 다시 “미인수” 상태로 되돌림

---

### 3-5. 분실물 단건 조회 및 삭제

| 메서드 | 경로 | 설명 | Request | Response |
|--------|------|------|---------|----------|
| GET | `/api/admin/lost-items/{id}` | 분실물 단건 조회 | 없음 | LostItemResponse |
| DELETE | `/api/admin/lost-items/{id}` | 분실물 삭제 (소프트 삭제) | 없음 | 204 No Content |

- **DELETE**는 DB에서 실제로 삭제하지 않고 `isActive=false` 로만 변경.  
  목록 API는 `isActive=true` 인 데이터만 내려주므로, 삭제 후에는 목록에 안 보임.

---

## 4. 프론트 개발 시 체크리스트

- [ ] 모든 `/api/admin/**` 요청에 `Authorization: Bearer {accessToken}` 헤더  
- [ ] 로그인/재발급/로그아웃 등 쿠키 쓰는 요청에 `credentials: 'include'` (또는 `withCredentials: true`)  
- [ ] Access Token은 메모리 등에만 보관 (localStorage 비권장)  
- [ ] 공지 목록에서 `isEmergency === true` 인 항목을 **맨 위**에 배치  
- [ ] 공지/분실물 목록 모두 `content` 배열 + `totalPages`, `number` 등으로 페이지네이션 처리  
- [ ] 에러 시 `status`, `message`, `errors` 로 메시지/필드 에러 표시

위 내용만 지키면 현재 구현된 백엔드와 그대로 맞춰서 **공지·분실물 관리자 프론트**를 만들 수 있습니다.
